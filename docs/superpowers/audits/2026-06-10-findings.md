Audit per plan `docs/superpowers/plans/2026-06-10-full-site-audit.md`. 10 parallel audit passes over all pages, hooks, edge functions, migrations; every Critical finding re-verified line-by-line against the code.

**Themes:** (1) Money/credit paths trust client input or non-atomic updates. (2) Several edge functions lack in-code auth. (5) i18n: marketing pages are bilingual and complete, but Events/News, Account, Dashboard, Club and several pages are hardcoded German.

## CRITICAL (all verified in code)

### C1. Double-booking protection never engages — customers can be charged without getting a slot
- **Where:** `supabase/migrations/20260104225807_*.sql:15-19`, `src/hooks/useBookingLocation.ts:181`, `supabase/functions/create-guest-booking/index.ts:162`, `supabase/functions/stripe-webhook/index.ts:219`
- **What:** The `no_overlapping_bookings` exclusion constraint only covers `status IN ('pending','confirmed')`, but the app creates every booking as `pending_payment` — a status the constraint ignores. Two users can hold the same slot and both pay; the second webhook's `UPDATE → confirmed` then violates the constraint, the error is only logged, and the booking stays `pending_payment` while the customer was charged.
- **Fix:** Recreate the constraint with `WHERE status IN ('pending_payment','confirmed')` (excluding expired holds) and surface webhook confirm failures (refund path).

### C2. Authenticated users can set their own booking price
- **Where:** `supabase/functions/create-checkout-session/index.ts:214` (`const totalPriceCents = booking.price_cents || priceCents;`), `src/hooks/useBookingLocation.ts:182`
- **What:** The booking row (incl. `price_cents`) is inserted directly from the browser; RLS only checks ownership, not price. The edge function recomputes the correct server-side price but then prefers the client-stored value. A tampered insert (`price_cents: 1`) flows straight into Stripe's `unit_amount` — pay 1 cent for any court.
- **Fix:** Always use the server-recomputed price (or reject when `booking.price_cents !== priceCents`).

### C3. Guest booking PII readable by anyone (anonymous enumeration)
- **Where:** `supabase/migrations/20260412200000_guest_bookings.sql:20-23`
- **What:** Policy `anon_select_guest_booking ... FOR SELECT TO anon USING (user_id IS NULL)` lets ANY anonymous client read EVERY guest booking — names, emails, phone numbers, times, locations. The anon key is public in the JS bundle.
- **Fix:** Drop the anon SELECT policy; serve guest-booking lookups through a service-role edge function keyed by booking UUID.

### C4. `rewards-claim` overwrites wallet balance with a wrong ledger sum — balances corrupt on every claim
- **Where:** `supabase/functions/rewards-claim/index.ts:72,133`; RPC in `20251222143818_*.sql:204-214`; `marketplace-redeem/index.ts` and `friends-api/index.ts` (no ledger writes)
- **What:** `get_user_rewards_balance` sums ALL `points_ledger` rows (including SKILL play credits), and the claim writes that absolute number into `wallets.reward_credits`. Marketplace redemptions and friend bonuses change the wallet without ledger entries — so any reward claim resurrects spent credits, wipes friend bonuses, and double-counts skill credits.
- **Fix:** Make wallet updates incremental and write a ledger row for every wallet mutation (single source of truth).

### C5. Any user can mint unlimited play credits via `/skill-credit`
- **Where:** `supabase/functions/p2g-points-api/index.ts:357-516`; exposed in `src/hooks/useP2GPoints.ts:381-394`
- **What:** The endpoint takes client-supplied `match_id` and `ai_score`/`manual_score` with no check that the match exists or belongs to the caller. Idempotency keys off the client-chosen `match_id`, so inventing new IDs mints credits indefinitely — inflating wallets, levels and rankings.
- **Fix:** Restrict to service-role/camera flow, or verify match ownership + completed booking server-side; remove the client mutation.

### C6. Referral rewards farmable without authentication
- **Where:** `supabase/functions/referral-api/index.ts:93-176`
- **What:** `POST /attribution` reads `{referralCode, referredUserId}` from the body with no auth check on this path (the platform `verify_jwt` gate is satisfied by the public anon key) and credits the referrer via service role. Anyone can farm referral rewards for their own code with fabricated UUIDs.
- **Fix:** Derive `referredUserId` from the caller's verified JWT and validate account age.

### C7. Club quota forgery — negative/false `duration` accepted
- **Where:** `supabase/functions/club-booking-api/index.ts:163-174,275,314-322`
- **What:** `duration` comes from the request body and is never compared to `endTime - startTime` nor checked positive (`!duration` blocks only 0). A club member can book a 2-hour slot for 1 minute of quota, or send a negative duration which passes `remaining < duration` and inserts a negative ledger row — permanently inflating the club's quota. Related: no `start < end` / max-length check lets one request block a court for weeks (see H-club below).
- **Fix:** Recompute duration server-side from start/end; require positive integer within a sane max.

### C8. Password reset is a dead end — users can never recover accounts
- **Where:** `src/pages/Auth.tsx:18,61-65` (mode `"reset"` set but never rendered); no `supabase.auth.updateUser` anywhere in src/
- **What:** The recovery email links to `/auth?mode=reset`, but the page only renders `login`/`register`/`forgot` forms. Worse, the recovery link logs the user in, and the role-redirect effect bounces them to `/account` before they could set a new password. No password is ever changed.
- **Fix:** Add a reset form calling `supabase.auth.updateUser({ password })` and suppress the auto-redirect while in reset mode.

## HIGH

### Payments / Booking
- **H1. Credits double-spend across parallel checkouts** — validated at session creation, deducted only in the webhook via non-atomic read-modify-write clamped with `max(0,…)`. Reserve credits atomically at session creation; refund on expiry. (`create-checkout-session:283-321`, `stripe-webhook:339-359`)
- **H2. Voucher soft-reserve leaks** — partial-discount vouchers increment `current_uses` at session creation but are only released on Stripe's `checkout.session.expired`; user-cancel and the auto-cancel cron never release, so limited vouchers get exhausted by abandoned checkouts. (`create-checkout-session:239-269`, migration `20260415120000`)
- **H3. Newsletter signup fails for every logged-out visitor** — public `/events` form inserts into `newsletter_subscribers`, but migration `20260402074051` restricts INSERT to `authenticated`. The form's main audience always gets an error toast. (`NewsletterCTA.tsx:22`)

### Backend / Edge functions
- **H4. translate-content has zero in-code auth** — callable with the public anon key; uses service role to overwrite `*_en` columns of 5 tables and burn DeepL quota. Also missing from `config.toml`. Add the admin check used by `generate-article`. (`translate-content/index.ts:99-216`)
- **H5. Integration secrets readable from the browser** — `site_integration_configs` SELECT policy returns full config (Stripe secret, webhook secret, Resend/Anthropic/DeepL keys) to any admin's browser; UI claims they're encrypted (false). Make reads server-side only / boolean flags. (`AdminIntegrations.tsx:139`, migration `20260413130000`)
- **H6. rewards-trigger trusts body `userId`** — admin-JWT callers can mint reward instances for arbitrary users; unaudited. (`rewards-trigger/index.ts:115-120`)
- **H7. camera-webhook scopes the API key to a location only on `start-session`** — `join-session`, `match-complete` (awards credits), `session-status` accept any session ID from any valid key. (`camera-webhook/index.ts:217-336,587-623`)

### Social / P2G
- **H8. Friend request can never be re-sent after decline** — `/request` falls through to INSERT for declined/cancelled rows → unique-constraint 500. Update the existing row back to pending. (`friends-api/index.ts:171-216`)
- **H9. Lobby invite acceptance bypasses capacity/status; join has a TOCTOU race** — full or cancelled lobbies can be over-filled. (`lobby-api/index.ts:469-509,1025-1063`)
- **H10. Two parallel daily-login bonus systems** — `rewards-api daily_login` and `p2g claim-daily` have separate dedupe stores; users can collect both daily bonuses with diverging streaks. Consolidate. (`DashboardRewards.tsx:89-107` vs `p2g-points-api:700-813`)
- **H11. lobby-api `get_lobby` leaks private lobbies + member profiles without auth.** (`lobby-api/index.ts:59-65,319-409`)
- **H12. DashboardHome bypasses friends-api** — direct table accept/decline skips +50 reward, grant-lock, and notification; decline-deletes diverge from API semantics. Reuse `useFriendships`. (`DashboardHome.tsx:149-173`)

### Frontend correctness
- **H13. Marketplace redemption never refreshes balances** — invalidates `["account-data"]`/`QUERY_KEYS.account`, but no query uses those keys (`useAccountData` isn't React Query). Headers, "Leistbar" badges, stock stay stale until reload. (`useMarketplaceRedeem.ts:52`)
- **H14. Broken nav targets:** `/app/marketplace`, `/app/rewards` (DashboardRewards CTAs), `/app/p2g-points` (camera notifications), `/friends?tab=requests` (friend notifications) — none of these routes exist; all land on 404. (`DashboardRewards.tsx:238,432`, `rewards-claim:145`, `friends-api:231`, `camera-webhook:533`)
- **H15. SkillLast5 field mismatch** — component reads `average_skill_level`/`play_credits`; API returns `avg_skill_level`/`play_credits_earned` → always "Ø 0.0" and empty credits. (`SkillLast5Section.tsx:63,111,127`)
- **H16. `?redirect=` ignored after login** — RequireAuth passes the deep link, Auth.tsx never reads it; every protected deep link lands on /account. Validate `startsWith("/") && !startsWith("//")` when implementing. (`Auth.tsx`, `RequireAuth.tsx:24`)
- **H17. Superadmin email bypass is client-only** — `fsteinfelder@padel2go.eu` passes `useAdminAuth` without a `user_roles` row, but nearly all RLS policies and admin edge functions check `has_role()`; the account sees the admin UI but mutations fail unless it also has the DB role. Seed the role via migration. (`useAdminAuth.ts:6,22`)
- **H18. Public profile isn't public** — `/u/:username` is a public route but `public-profile-api` 401s without a logged-in user; logged-out visitors get "user doesn't exist" for every shared link. (`public-profile-api:44-60`)
- **H19. TouchpointCarousel index crash** — fallback slides (5) auto-advance; when fewer DB slides load, `items[current]` dereferences out of range → page crash to error boundary. Clamp the index. (`TouchpointCarousel.tsx:25-66`)

### Club portal / UI
- **H20. `allowed_booking_windows` evaluated in UTC** — Deno runs UTC; windows configured in German local time → 1–2h offset, weekday flips near midnight. (`club-booking-api:225-248`)
- **H21. ClubCourtFeatures switches double-toggle** — Card `onClick` + nested Switch `onCheckedChange` both fire; clicking the switch nets no change but marks "unsaved changes". (`ClubCourtFeatures.tsx:223-345`)
- **H22. i18n `returnEmptyString: false` breaks intentionally-empty values** — raw keys ("legalNotice", "partners.titlePart2") render on German legal pages/home; German suffixes ("Uhr", "trifft", "öffnen") leak into the English checkout and partner pages.
